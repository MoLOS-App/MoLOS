## 🎉 Welcome to MoLOS v1.0.0!

This marks the first major stable release of MoLOS (Modular Life Organization System). After months of development and refinement, we're proud to deliver a production-ready, deeply modular productivity suite that puts privacy and flexibility first.

## 📋 Executive Summary

MoLOS v1.0.0 introduces:

- ✅ **9 production-ready modules** covering tasks, AI, goals, health, finance, meals, markdown, and Google integration
- ✅ **Robust database migration system** (v2.1) with automatic backups and rollback support
- ✅ **Multi-LLM Council** for diverse AI perspectives and decision support
- ✅ **Markdown Editor** with rich editing, live preview, and export options
- ✅ **Production build system** with build-time module resolution
- ✅ **Svelte 5 with runes** for modern, reactive components
- ✅ **Privacy-first architecture** - self-hostable, no cloud dependencies for core functionality

## 🆕 New Features

### Core System

#### Production Build System

- **Build-time module resolution** - Modules are fetched and linked during Docker build
- **Deterministic builds** - Same modules, same version, every time
- **Smaller production images** - No runtime module fetching code
- **Enhanced security** - No dynamic module execution in production

#### Database Migration System v2.1

- **Automatic backups** with integrity verification before each migration
- **Transaction-wrapped migrations** - No partial states on failure
- **Reversible migrations** - Rollback support with `.down.sql` files
- **Migration locking** - Prevents concurrent migration runs
- **Checksum validation** - Detects modified migration files
- **Backup management** - Keep last 10 backups with automatic rotation
- **Interactive restoration** with confirmation prompts
- **Error categorization** - 9 specific error types for debugging

#### Module System

- **Monorepo workspaces** - All modules in `modules/` directory
- **Auto-discovery** - Modules automatically discovered by dev server
- **Workspace dependencies** - Use `@molos/module-{name}` with `workspace:*`
- **Simplified module sync** - Single `bun run dev` command for everything

### New Modules

- **Rich Markdown Editor** - Clean, intuitive editing experience
- **Live Preview** - Real-time markdown rendering as you type

- **Search & Navigation** - Quick document search and table of contents
- **Custom Themes** - Dark, light, and custom editor themes

#### MoLOS-LLM-Council v1.0.0

- **Multi-LLM consultation** - Query multiple LLMs simultaneously
- **Persona management** - Create and manage AI personas with different expertise
- **Conversation history** - Track and review past council consultations
- **Voting system** - Rate and select best LLM responses

#### MoLOS-Tasks v1.0.4

- **Project management** - Organize tasks into projects with milestones
- **Area management** - Group related projects by life areas (Work, Personal, etc.)
- **Daily log** - Track daily task completion and progress

## 🔄 Breaking Changes

### Module Installation

**Before** (pre-v1.0.0):

```bash
bun add @molos/module-tasks@github:MoLOS-App/MoLOS-Tasks
```

**After** (v1.0.0):

```bash
# Modules now live in monorepo - just add to modules/ directory
cp -r path/to/MoLOS-Module modules/
bun run dev  # Auto-discovers and syncs
```

**Impact**: External modules are no longer installed via npm in development. They're developed in the monorepo `modules/` directory.

### Import Patterns

**Before** (pre-v1.0.0):

```typescript
// Fragile relative paths
import { TaskRepository } from '../../../../../server/repositories/task-repository.js';
```

**After** (v1.0.0):

```typescript
// Use $module alias for module-internal imports
import { TaskRepository } from '$module/server/repositories/task-repository.js';

// Use $lib alias for main app imports
import { db } from '$lib/server/db';
```

**Impact**: Module code must use `$module` alias for internal imports instead of relative paths. This ensures imports work in both development and production builds.

### Database Migration Commands

**Before** (pre-v1.0.0):

```bash
bun run db:generate  # Auto-generated migrations
```

**After** (v1.0.0):

```bash
# FORBIDDEN - drizzle-kit generate removed for safety
# bun run db:generate  # ERROR: Command not found

# Instead, create migrations manually with descriptive names
bun run db:migration:create --name add_feature --module MoLOS-Tasks --reversible

# Apply migrations safely with automatic backups
bun run db:migrate  # Now an alias for db:migrate:improved
```

**Impact**: `drizzle-kit generate` is banned. Migrations must be created manually using `db:migration:create`. This prevents schema desync and enforces naming conventions.

## ⚠️ Deprecations

### external_modules/ Directory

The `external_modules/` directory has been removed. All modules now live in the `modules/` directory as monorepo workspaces.

### Legacy Import Paths

Relative imports like `../../server/repositories/file.js` are deprecated. Always use `$module` alias for module-internal code.

## 📊 Known Issues

### Minor Issues

1. **Module hot-reload** - Changing module config requires full dev server restart
2. **Large datasets** - SQLite performance may degrade with >100k rows per table (planned PostgreSQL migration path)
3. **AI rate limits** - Some LLM providers may have rate limiting that affects council features

- Redis caching layer for performance
- PostgreSQL migration option for high-volume deployments

## 📥 Download & Installation

### Quick Start

```bash
# Clone repository
git clone https://github.com/MoLOS-org/MoLOS.git
cd MoLOS

# Install dependencies
bun install

# Start development server (auto-discovers modules)
bun run dev
```

### Production Deployment

```bash
# Build production image
docker build -t molos:latest .

# Run container
docker run -d -p 4173:4173 -v ./data:/data --name molos molos:latest
```

### Module Management

All modules are included in the official Docker image. Activate/deactivate modules via the UI settings page.
