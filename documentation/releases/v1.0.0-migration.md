# MoLOS v1.0.0 Migration Guide

> **For**: Developers upgrading from pre-v1.0.0 versions
> **Updated**: 2026-03-17
> **Status**: ✅ Complete

---

## Overview

This guide helps you upgrade your MoLOS installation and custom modules from pre-v1.0.0 versions to v1.0.0.

**What changed in v1.0.0:**

- Modules moved from `external_modules/` to monorepo `modules/` directory
- Database migration system overhauled (v2.1) with manual migration creation
- Import patterns changed (`$module` alias required)
- Production build system now uses build-time module resolution

---

## 📋 Pre-Migration Checklist

Before starting migration, verify:

- [ ] You have a backup of your current database: `cp data/molos.db data/molos.db.backup`
- [ ] You have a backup of custom modules if any
- [ ] You're running Node.js >= 18 or Bun >= 1.0.0
- [ ] You've read the [v1.0.0 Release Notes](./v1.0.0.md)
- [ ] You've noted any custom modifications to core files

## Step 1: Update Repository

```bash
# Backup your current state
git branch backup-before-v1-upgrade
cp data/molos.db data/molos.db.backup

# Pull latest changes
git fetch origin
git checkout v1.0.0
```

---

## Step 2: Update Dependencies

```bash
# Update package manager
bun install

# Verify all packages installed correctly
bun check  # Should show no errors
```

## Step 3: Module Migration

### If You Use Official Modules

No action needed - all official modules are in the monorepo `modules/` directory.

```bash
# Verify modules are present
ls modules/
# Expected: ai, MoLOS-Tasks, MoLOS-AI-Knowledge, MoLOS-LLM-Council, etc.
```

### If You Have Custom Modules

**Before** (pre-v1.0.0):

```
MoLOS/
└── external_modules/
    └── MyCustomModule/
        ├── package.json
        ├── src/
        └── drizzle/
```

**After** (v1.0.0):

```
MoLOS/
└── modules/
    └── MyCustomModule/
        ├── package.json
        ├── src/
        └── drizzle/
```

**Migration Steps:**

```bash
# 1. Move your custom module from external_modules to modules
mv external_modules/MyCustomModule modules/

# 2. Update package.json in your module
cd modules/MyCustomModule

# 3. Change package name if needed (optional)
# Before: "name": "my-custom-module"
# After: "name": "@molos/module-my-custom-module" (recommended)

# 4. Update imports to use $module alias
# See Step 4 below for details

# 5. Return to root
cd ../..
```

## Step 4: Update Import Patterns

### Module-Internal Imports

**Before** (pre-v1.0.0):

```typescript
// Fragile relative paths
import { TaskRepository } from '../../../../../server/repositories/task-repository.js';
import { TaskStatus } from '../../../../models/index.js';
```

**After** (v1.0.0):

```typescript
// ✅ Use $module alias for module-internal imports
import { TaskRepository } from '$module/server/repositories/task-repository.js';
import { TaskStatus } from '$module/models/index.js';
import { mySchema } from '$module/server/database/schema.js';
```

**Why**: The `$module` alias is resolved by a custom Vite plugin that detects which module is importing and resolves to that module's `src/` directory. This ensures imports work in both development and production builds.

### Main App Imports

**Before** (pre-v1.0.0):

```typescript
// Sometimes used relative paths
import { db } from '../../../../../src/lib/server/db';
```

**After** (v1.0.0):

```typescript
// ✅ Always use $lib alias for main app imports
import { db } from '$lib/server/db';
import { Button } from '$lib/components/ui/button';
import { auth } from '$lib/server/auth';
```

**Why**: The `$lib` alias points to `src/lib/` regardless of where the import originates.

### Cross-Module Imports

If you need to import from another module (rare, usually avoid):

```typescript
// Pattern 1: Import from module's lib directory
import { utility } from '$lib/modules/MoLOS-OtherModule/lib/utils.js';

// Pattern 2: Import specific subdirectory
import { OtherType } from '$lib/models/external_modules/MoLOS-OtherModule';
```

## Step 5: Update Database Migrations

### Migration Command Changes

**Before** (pre-v1.0.0):

```bash
# Auto-generate migrations (FORBIDDEN in v1.0.0)
bun run db:generate
```

**After** (v1.0.0):

```bash
# Create migration manually with descriptive name
bun run db:migration:create --name add_new_feature --module MoLOS-MyModule --reversible

# Edit the generated SQL file
vim modules/MoLOS-MyModule/drizzle/0001_add_new_feature.sql
vim modules/MoLOS-MyModule/drizzle/0001_add_new_feature.down.sql

# Apply migrations safely (auto-backup + transaction)
bun run db:migrate
```

**Why**: `drizzle-kit generate` is banned because it:

- Creates journal/SQL desync (migrations in journal but no SQL files)
- Generates random names with zero context
- Cannot enforce table naming conventions (`MoLOS-{Name}_{table}`)
- Overwrites manual migrations without warning

### Migration File Structure

**New** (v1.0.0) - Reversible migrations:

```sql
-- modules/MoLOS-MyModule/drizzle/0001_add_feature.sql
-- Migration: add_feature
-- Module: MoLOS-MyModule
-- Created: 2026-03-17

CREATE TABLE MoLOS-MyModule_items (
    id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    name TEXT NOT NULL
);

--> statement-breakpoint

CREATE INDEX idx_my_module_items_user_id ON MoLOS-MyModule_items(user_id);
```

```sql
-- modules/MoLOS-MyModule/drizzle/0001_add_feature.down.sql
-- Rollback: add_feature
-- Module: MoLOS-MyModule
-- Created: 2026-03-17

DROP INDEX IF EXISTS idx_my_module_items_user_id;

--> statement-breakpoint

DROP TABLE IF EXISTS MoLOS-MyModule_items;
```

## Step 6: Update Component Props

### Svelte 5 Runes

**Before** (pre-v1.0.0 - Svelte 4 stores):

```typescript
import { writable } from 'svelte/store';

export let count = writable(0);
export let name = '';
```

**After** (v1.0.0 - Svelte 5 runes):

```typescript
// ✅ Use $state for reactive state
let count = $state(0);
let name = $state('');

// ✅ Use $props for component props
interface Props {
	value: number;
	onChange?: (value: number) => void;
}

let { value, onChange }: Props = $props();

// ✅ Use $bindable for two-way bindings
let bound = $bindable(value);

// ✅ Use $derived for computed values
let doubled = $derived(() => bound * 2);

// ✅ Use .svelte.ts extension for shared state
// store.svelte.ts
let sharedCount = $state(0);
export const store = {
	get value() {
		return sharedCount;
	}
};
```

**Exception**: `export let data` is still valid for passing data from `+page.server.ts`/`+layout.server.ts` load functions.

## Step 7: Run Database Migration

```bash
# Check migration status
bun run db:repair

# Apply all pending migrations (safe with auto-backup)
bun run db:migrate

# Validate schema integrity
bun run db:validate
```

## 🐛 Common Migration Issues

### Issue: "Could not resolve '$module/server/...'"

**Cause**: Not using `$module` alias for module-internal imports

**Solution**:

```typescript
// ❌ Wrong
import { repo } from '../server/repositories/file.js';

// ✅ Correct
import { repo } from '$module/server/repositories/file.js';
```

### Issue: "Module not appearing in sidebar"

**Cause**: Module not discovered or sync not run

**Solution**:

```bash
# Sync modules
bun run module:sync

# Check config.ts exists
ls modules/YourModule/src/config.ts

# Check environment filter
echo $VITE_MOLOS_AUTOLOAD_MODULES  # Should be empty or include your module
```

### Issue: "Checksum mismatch" on migrations

**Cause**: Migration file was modified after being applied

**Solution**:

```bash
# Option 1: Revert changes
git checkout modules/YourModule/drizzle/0001_name.sql

# Option 2: Create new migration
bun run db:migration:create --name fix_name_v2 --module YourModule
```

### Issue: "Table not found" for module tables

**Cause**: Migrations not applied or wrong table prefix

**Solution**:

```bash
# Check table name follows convention
# Correct: MoLOS-MyModule_table_name
# Wrong: MyModule_table_name (missing prefix)
# Wrong: MoLOS_MyModule_table_name (underscores in module ID)

# Apply migrations
bun run db:migrate

# Verify table exists
sqlite3 data/molos.db ".tables MoLOS-%"
```

## 📊 Post-Migration Checklist

After completing migration, verify:

- [ ] All expected modules appear in sidebar
- [ ] Database tables follow naming convention (`MoLOS-{Name}_{table}`)
- [ ] No import errors in dev server console
- [ ] Module routes load correctly (no 404s)
- [ ] AI tools are accessible via assistant
- [ ] Migrations applied successfully
- [ ] Production build completes without errors (if deploying)

---

## 🔄 Rollback Procedure

If migration fails:

```bash
# Stop application
pkill -f "vite|node"

# Restore database from backup
cp data/molos.db.backup data/molos.db

# Rollback code changes
git checkout backup-before-v1-upgrade

# Reinstall old dependencies
bun install

# Restart
bun run dev
```
